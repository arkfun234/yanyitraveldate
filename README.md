# 福井県観光AI推薦システム（研究プロトタイプ）

## 1. 研究概要

本プロジェクトは、福井県観光における旅行者タイプ診断と推薦根拠の提示を重視したAI推薦システムの研究プロトタイプである。

従来のAI旅行推薦では、推薦結果そのものは提示される一方で、「なぜその場所が推薦されたのか」という根拠がユーザーに伝わりにくい場合がある。また、過去データに基づく推薦では、出現頻度の高い観光地に推薦が偏りやすいという課題もある。本システムでは、12問の旅行スタイル問診を用いてユーザーの旅行傾向を推定し、旅行者画像、推薦プロセス、過去推薦傾向データ、宿泊施設データを可視化することで、推薦結果の理解可能性を高めることを目的とする。

さらに、推薦エリアを単純な上位3件として提示するのではなく、「定番推薦」「個性化推薦」「発見推薦」の3種類に分けて提示することで、推薦結果の安定性、個人適合性、多様性のバランスを取ることを目指す。

## 2. 研究目的

本研究の目的は、単に観光地を推薦するだけではなく、推薦結果の生成過程と根拠を提示することで、ユーザーが推薦内容を理解しやすい観光推薦システムを構築することである。

特に以下の点を重視する。

- 旅行者タイプの可視化
- 推薦根拠の明示
- 過去推薦データに基づく推薦傾向の提示
- 宿泊施設データとの連携
- AI生成旅行プランにおける推薦候補の制御
- 推薦結果における人気スポット偏重の抑制
- 定番性・個人適合性・発見性を考慮した推薦構成

## 3. 使用データ

本システムでは、主に以下のデータを利用する。

| ファイル名 | 内容 | 用途 |
|---|---|---|
| `questions12.json` | 12問の旅行スタイル質問 | ユーザー回答の取得 |
| `questions12_to_psych144.json` | 回答選択肢から心理・行動タグへの対応表 | 旅行傾向のタグ化 |
| `psych144_to_53_weights.fixed-new.json` | 心理・行動タグから53次元特徴量への重み | 行動特徴量の生成 |
| `cluster_profiles.json` | 旅行者クラスター情報 | 類似旅行者タイプの照合 |
| `20_top_places.json` | クラスター別代表的観光エリア | 推薦候補の補完 |
| `spot_weights_by_companion_season.json` | 同行者・旅行時期別推薦傾向 | 観光エリア推薦 |
| `hotels.json` | QR設置エリアに基づく宿泊施設データ | 宿泊施設推薦 |
| `data_summary_2025.json` | 2025年度福井県観光AI推薦ログの要約 | 推薦データ根拠の表示・定番推薦の抽出 |

## 4. 推薦プロセス

本システムでは、AIに直接推薦を任せるのではなく、以下の段階的な処理によって推薦結果を生成する。

1. 12問の回答を取得
2. 回答を心理・行動タグに変換
3. タグ重みに基づいて53次元特徴量へ変換
4. 旅行者クラスターと照合
5. 同行者・旅行時期別の過去推薦傾向を参照
6. 推薦エリア・宿泊施設・AI旅行プランを提示

この処理により、ユーザーに対して推薦結果だけでなく、推薦が生成される流れも提示できる。

## 5. 推薦多様性を考慮した推薦手法

本システムでは、推薦スコア上位3件をそのまま提示するのではなく、推薦エリアを以下の3種類に分けて提示する。

| 推薦分類 | 抽出方法 | 目的 |
|---|---|---|
| 定番推薦 | 2025年度推薦ログで高頻度に出現した観光エリアを抽出 | 多くのユーザーに共通して推薦されやすい安定候補を提示する |
| 個性化推薦 | 同行者・旅行時期別の推薦傾向、および旅行者タイプとの対応に基づいて抽出 | ユーザー条件に適合した候補を提示する |
| 発見推薦 | 高頻度スポットに偏りすぎないよう、旅行者クラスターとの関連性や中位候補を考慮して補完 | 推薦結果に多様性と発見性を持たせる |

この構成により、推薦結果を「人気が高く安定した候補」「ユーザー条件に合う候補」「新たな発見につながる候補」として分離し、観光推薦における人気スポット偏重を抑制することを目指す。

### 推薦分類の考え方

- **定番推薦**：推薦ログ全体で出現頻度が高い観光エリアを用いる。初めて福井を訪れるユーザーにとっても受け入れやすい安定候補として扱う。
- **個性化推薦**：同行者、旅行時期、旅行者タイプとの対応を考慮し、ユーザー条件に近い候補を提示する。
- **発見推薦**：上位人気スポットだけに推薦が集中しないよう、クラスター別代表エリアや中位候補を利用し、推薦の幅を広げる。

この手法は、推薦の正確性のみを追求するのではなく、観光推薦における利用者の納得感と探索性を高めるための設計である。

## 6. 本システムの特徴

本システムの特徴は、推薦結果の説明可能性と推薦多様性を重視している点にある。

| 比較項目 | 一般的なAI推薦 | 本システム |
|---|---|---|
| 推薦エリア提示 | あり | あり |
| 旅行者タイプ可視化 | 不明確になりやすい | あり |
| 推薦理由提示 | 不十分な場合がある | あり |
| データ根拠表示 | 少ない | あり |
| 宿泊施設データ連携 | 場合による | あり |
| 推薦多様性 | 人気スポットに偏る可能性がある | 定番・個性化・発見の3分類で提示 |
| 研究利用 | 結果提示中心 | 説明可能性と多様性を重視 |

## 7. AI旅行プラン生成について

AI旅行プラン生成では、システムが抽出した推薦エリアのみを利用するように制御している。これにより、AIが独自に観光地を追加することを抑制し、推薦結果とAI生成文の整合性を保つ。

また、営業時間、料金、交通状況などの変動情報については断定しないようにし、実際の旅行前に確認が必要であることを明示する。

## 8. 現在の研究上の位置づけ

## 12. ローカルAzure OpenAI旅行プラン生成

この機能はローカル開発・デモ専用です。Azure OpenAIの設定はプロジェクトルートの
`.env`だけに保存し、フロントエンドやGitHub Pagesでは使用しません。

Step 1:

```powershell
py -m pip install -r requirements-local-ai.txt
```

Step 2:

プロジェクトルートにAzure OpenAI設定を含む`.env`を作成する。

Step 3:

```powershell
py scripts\local_ai_server.py
```

Step 4:

```powershell
py -m http.server 5500
```

Step 5:

ブラウザで以下を開く。

```text
http://localhost:5500
```

アンケートに回答し、推薦結果の比較テストで
「AI旅行プランを生成（ローカルAPI）」をクリックする。
